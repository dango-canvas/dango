# Dango 项目协作备忘录

## 核心理念

此备忘录旨在同步我们协作的底层逻辑与偏好。我们的目标是构建一个体验一致、代码轻量且易于维护的产品。请将以下内容作为我们共同决策的思考框架。

## 设计与交互原则

*   **全局一致性**
    我们的交互设计追求一种整体感。因此，我们倾向于保持这种**逻辑一致性**，以维护统一的用户感知。如果发现有相关样式被注释，通常是为保留此全局行为的有意为之。

*   **环境适应性**
    元素的视觉表现应与所在上下文融合。例如，链接文本的颜色会继承其父节点，仅保留下划线作为标识。这种设计选择旨在减少视觉干扰，使信息层级更清晰。

*   **注重实现细节**
    我们关注那些影响最终体验的微小细节。例如，确保元素的边框不被 `overflow: hidden` 意外裁切，或是在DOM状态切换（如进入 `contenteditable` 模式）时避免布局抖动和不必要的节点残留。

## 开发哲学

*   **轻量化与独立性**
    项目的一个目标是实现单文件分发。我们优先采用原生 API 或轻量级实现（例如，使用正则表达式处理简单的语法高亮），审慎引入外部依赖，以控制最终产物体积和复杂性。

*   **局部化与精确修改**
    我们倾向于进行目标明确、范围可控的代码修改。这有助于保持代码结构的稳定性和可预测性，减少大规模重构可能带来的未知风险。

*   **注释的意图性**
    代码中的注释，包括被注释掉的代码块，均被视为有意的记录。它们可能包含历史决策的上下文、被搁置的备选方案或未来探索的思路。除非我们明确讨论并决定移除，否则应予以保留。

## 协作模式

*   **文档同步与规范对齐**
    在更新技术规范（如 `SPEC.md`）时，需要尊重并保持原有的逻辑流和层级结构。新增特性的描述应当与现有同类功能（例如“Todo 列表”、“图片节点”）平级，并精准仿照现有的格式颗粒度（如详述样式、交互等维度）进行无缝增补，避免生硬打乱原有的目录和上下文。

*   **前瞻性分析**
    在实施一项变更前，我们习惯于先推演其潜在的连锁影响。例如：一个 DOM 结构的调整，是否会影响到纯文本内容的序列化逻辑？一个 CSS 层级的变动，是否会在选中或编辑状态下产生非预期的遮挡？

*   **以最终决策为准**
    我们鼓励提出多种技术方案和优化建议。这是一个开放的探讨过程。然而，一旦用户做出明确的修改（无论是代码实现、目录结构还是注释内容），该决策即成为我们后续工作的基准。我们应在此基础上继续推进，而不是反复修正已确认的选择。
